## 2. 背景与目标

### 2.1 背景

当前企业普遍面临以下数据管理挑战：

1. **数据孤岛严重** — Oracle/SQL Server/国产数据库分散，跨库查询困难
2. **SQL 门槛高** — 业务人员无法自主配置数据流程，依赖技术人员
3. **ETL 流程僵化** — 传统 ETL 配置复杂，不够灵活，响应慢
4. **AI 语义歧义** — "客户"等术语在不同表中有不同含义，NL→SQL 易出错
5. **数据质量失控** — ETL 过程中异常数据无法自动检测
6. **指标散落** — 同一指标（如"收入"）在各流程/报表中定义不一致，数据不可信
7. **修正不沉淀** — 用户修正 AI 生成的 SQL 后，规则无法复用

### 2.2 目标

**核心价值：** 把"写 SQL / 写 ETL 代码"变成"说人话配置数据流程"

**业务目标：**
- 业务人员无需 SQL 基础，自然语言即可配置数据流程
- AI 生成查询语句可正确执行，结果符合预期
- 多数据源查询稳定，支持异常告警和重试
- 全系统所有报表/AI 查询同一指标，数值一致
- AI 能从用户修正中学习，修正规则自动沉淀到知识库

### 2.3 成功指标

| 维度 | 指标 | 目标值 |
|------|------|--------|
| 易用性 | 业务人员自助配置率 | ≥ 80% |
| 查询准确率 | AI 生成 SQL 首版正确率 | ≥ 85% |
| 稳定性 | 多数据源查询成功率 | ≥ 99% |
| 报表多样化 | 支持图表类型数量 | ≥ 8 种 |
| 质量门禁 | 质量问题阻断率 | 100% |
| 版本可控 | 变更可追溯率 | 100% |
| 指标一致性 | 跨报表指标一致率 | 100% |
| 语义演进 | 规则复用率（同类查询应用修正规则比例） | ≥ 70% |

## 3. 用户场景

### 3.1 目标用户

| 角色 | 职责 | 用户画像 |
|------|------|---------|
| **管理员** | 系统配置、数据源管理、用户管理 | IT/数据团队负责人 |
| **数据分析师** | 创建/管理数据流程、执行查询、分析数据 | 专业数据分析人员 |
| **业务人员** | 配置简单数据流程、消费数据报表 | 业务部门运营/市场人员 |
| **运维** | 系统监控、部署维护 | 基础设施团队 |

### 3.2 用户故事

作为 **业务人员**，我希望 **用自然语言描述我的数据需求**，以便 **无需编写 SQL 即可获取数据分析结果**。

作为 **数据分析师**，我希望 **配置可复用的数据流程**，以便 **提高数据分析效率，减少重复工作**。

作为 **管理员**，我希望 **管控数据访问权限**，以便 **在提升数据价值的同时保障数据安全**。

作为 **财务人员**，我希望 **全系统同一指标数值一致**，以便 **不同报表的数据可以交叉验证**。

### 3.3 核心使用场景

**场景1：业务人员自助分析**
- 触发条件：业务人员登录系统，用自然语言提问"今年各区域销售额是多少？"
- 用户行为：系统解析意图 → 生成 SQL → 执行查询 → 返回图表
- 预期结果：自动生成折线图/饼图展示各区域销售额分布

**场景2：数据分析师配置 ETL 流程**
- 触发条件：数据分析师创建新数据流程
- 用户行为：选择数据源 → 配置清洗规则 → 设置质量门禁 → 保存流程
- 预期结果：流程自动执行，数据质量达标后进入下游

**场景3：指标一致性校验**
- 触发条件：AI 生成 SQL 查询"收入"
- 用户行为：系统通过语义层解析"收入"定义 → 自动应用指标公式
- 预期结果：查询结果与报表中"收入"指标数值完全一致

## 8. 风险评估

| 风险类型 | 风险描述 | 影响程度 | 应对措施 |
|---------|---------|---------|---------|
| 技术风险 | AI 生成 SQL 准确率不足，导致数据错误 | 高 | 设置置信度阈值，低于阈值强制人工确认；RLHF 持续优化 |
| 技术风险 | 跨库 join 性能问题 | 中 | 下推优先原则；超时熔断；结果集大小限制 |
| 业务风险 | 语义层配置成本高，企业不愿投入 | 中 | 提供默认模板；AI 自动推荐映射关系 |
| 业务风险 | 指标口径统一需要跨部门协调 | 中 | 自上而下推动；指标变更需评审流程 |
| 数据风险 | 质量门禁误阻断导致流程无法运行 | 中 | 支持白名单豁免；分级告警而非直接阻断 |
| 安全风险 | 敏感字段权限管控不到位导致数据泄露 | 高 | 列级加密；权限审计；最小权限原则 |

## 10. 技术架构概要

### 10.1 技术选型

| 层级 | 技术 |
|------|------|
| 后端 | Spring Boot + Spring AI Alibaba admin + Spring Alibaba Agent Framework |
| 前端 | React + ECharts |
| 数据库 | 国产数据库（达梦/人大金仓/PolarDB）+ SQL Server + Oracle + MySQL/PostgreSQL |
| 中间件 | Redis + RocketMQ |
| 部署 | 私有化部署 + 云原生/K8S |

### 10.2 架构原则

- **视图优先，物化为辅**：默认使用视图层，仅在性能需要时物化
- **下推优先，结果拉回**：跨库 join 优先下推到数据源执行
- **质量门禁前移**：每层 ETL 后置质量检查，不达标则阻断
- **语义层中心化**：指标定义统一管理，全系统共享

## 11. 边界定义

### 11.1 做的事项

- NL→SQL 生成与执行
- 多数据源接入与跨库查询
- 数据 ETL 流程配置与自动化
- AI 语义层（术语映射 + 知识记忆 + 歧义消解）
- 数据质量门禁
- 语义模型层（指标中心化）
- 自带报表能力

### 11.2 不做的事项

- ❌ 复杂机器学习建模（预测分析仅限简单统计预测）
- ❌ 全公司级数据治理/元数据管理
- ❌ 实时流式数据处理（仅支持批量 ETL，架构预留事件驱动扩展）
- ❌ 第三方 BI 工具集成（自带报表，语义层通过 API 导出）
- ❌ 复杂数据质量规则引擎（基础清洗过滤为主，复杂规则通过 Connector SDK 扩展）
- ❌ SSO 自定义 UI 改造（仅支持标准 SAML/OIDC/Oauth2 协议）
