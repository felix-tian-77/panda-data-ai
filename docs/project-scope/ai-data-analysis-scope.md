# AI 数据分析系统 - 需求大纲

## 1. 项目概述

**AI 数据分析系统** — 一个支持多数据源接入，自然语言驱动的数据 ETL 流水线平台。用户通过 AI 对话配置数据流程，实现数据采集 → 清洗过滤 → 分析报表的全链路自动化。

**核心价值:** 把"写 SQL / 写 ETL 代码"变成"说人话配置数据流程"

## 2. 技术栈

| 层级 | 技术选型 |
|------|---------|
| **后端** | Spring Boot + Spring AI Alibaba admin (https://github.com/spring-ai-alibaba/spring-ai-alibaba-admin) + Spring Alibaba Agent Framwork (https://github.com/alibaba/spring-ai-alibaba/tree/main/spring-ai-alibaba-agent-framework) |
| **前端** | React 技术栈 + ECharts |
| **数据库** | 国产数据库（达梦/人大金仓/PolarDB）+ SQL Server + Oracle + MySQL/PostgreSQL |
| **中间件** | Redis + RocketMQ |
| **部署** | 私有化部署 + 云原生/K8S |
| **认证** | 用户密码登录 + SSO 单点登录（支持 SAML/OIDC/Oauth2） |

## 3. 核心用户与权限

### 3.1 目标用户

| 角色 | 职责 |
|------|------|
| **管理员** | 系统配置、数据源管理、用户管理 |
| **数据分析师** | 创建/管理数据流程、执行查询、分析数据 |
| **业务人员** | 配置简单数据流程、消费数据报表 |
| **运维** | 系统监控、部署维护 |

### 3.2 权限模型

采用 RBAC（基于角色的访问控制）方式，按角色设置功能权限。

### 3.3 数据权限

支持**行级**和**列级**数据访问控制：

| 功能 | 描述 |
|------|------|
| 行级数据权限 | 按条件过滤数据（如：部门只能看本部门数据） |
| 列级数据权限 | 按字段脱敏/隐藏（如：薪资字段仅管理员可见） |
| 数据权限规则 | 支持配置基于用户属性/组织架构的动态数据过滤 |

## 4. 成功标准

| 维度 | 指标 |
|------|------|
| **易用性** | 业务人员无需 SQL 基础，自然语言即可配置数据流程 |
| **查询准确率** | AI 生成查询语句可正确执行，结果符合预期 |
| **稳定性** | 多数据源查询稳定，支持异常告警和重试 |
| **报表多样化** | 支持折线图/饼图/漏斗图/地图等 ECharts 图表类型 |
| **扩展性** | 支持配置 N 个数据流程，流程可复用/可嵌套 |
| **质量门禁** | 数据质量评分达标后才进入下游流程 |
| **版本可控** | 所有变更可追溯、可对比、可一键回滚 |
| **指标一致性** | 全系统所有报表/AI 查询同一指标，数值一致 |
| **语义可演进** | AI 能从用户修正中学习，修正规则自动沉淀到知识库 |

## 5. 要解决的问题

1. **数据孤岛** — 多数据源（Oracle/SQL Server/国产库）分散，跨库查询困难
2. **SQL 门槛高** — 业务人员无法自主配置数据流程，依赖技术人员
3. **ETL 流程僵化** — 传统 ETL 流程配置复杂，不够灵活
4. **报表依赖 BI 工具** — 数据到可视化之间的链路长，响应慢
5. **AI 语义歧义** — "客户"等业务术语在不同表中有不同含义，NL→SQL 易混淆
6. **数据质量失控** — ETL 过程中异常数据无法自动检测，质量问题在下游才暴露
7. **指标散落** — 同一指标（如"收入"）在各流程/报表中定义不一致，数据不可信
8. **修正不沉淀** — 用户修正 AI 生成的 SQL 后，规则无法复用到同类查询

| 分析类型 | 描述 |

## 8. 不做的事（边界）

- ❌ 不做复杂机器学习建模（预测分析仅限简单统计预测）
- ❌ 不做数据治理/元数据管理（仅做当前系统的数据字典，不做全公司级元数据治理）
- ❌ 不做实时流式数据处理（仅支持批量 ETL，但架构上预留事件驱动扩展能力）
- ❌ 不做第三方 BI 工具集成（自带报表能力，语义层仅通过 API 导出指标定义）
- ❌ 不做复杂数据质量规则引擎（基础清洗过滤为主，复杂规则通过 Connector SDK 扩展）
- ❌ SSO 集成不含自定义 UI 改造（仅支持标准 SAML/OIDC 协议对接）

## 9. 架构设计要点

### 9.1 中间表策略：视图优先，物化为辅

- 默认使用视图（View）层，仅在性能需要时物化中间结果
- 避免中间表膨胀，减少存储成本和数据同步复杂度
- 参考：Databricks lakehouse 视图层架构

### 9.2 跨库查询：下推优先，结果拉回

- 多数据源跨库 join 时，优先将查询下推到数据源执行（Federated Query）
- 仅拉取结果到中间层，避免大量原始数据跨网络传输
- 对于不支持下推的数据源，走应用层 join

### 9.3 事件驱动架构预留

- 触发器支持"事件触发"模式（用户上传文件 / 外部 API 调用 / RabbitMQ 消息）
- 定时触发与事件触发可混合配置，一条流程可以同时支持两种启动方式
- RocketMQ 作为事件总线，支持未来向实时流处理演进（不推翻实时处理边界限制）

### 9.4 质量门禁机制

- 每层 ETL 完成后自动执行质量检查（内置测试用例 + 质量评分）
- 评分低于阈值时，流程自动中断并告警，阻止脏数据进入下游
- 支持配置跳过质量门禁（仅限特定管理员权限）

### 9.5 AI 语义记忆架构

- 机构知识库：用户修正 SQL 时自动提取规则，存储为结构化知识条目
- RAG 检索：NL 查询时同时检索术语映射和历史修正记录，取相关性最高的 Top-K 作为上下文
- 表使用权重：AI 自动记录高频使用表/字段，优化表选择顺序和 NL→SQL 准确率

### 9.6 语义模型中心化架构

- 指标定义存储在语义模型服务中，而非散落在各 ETL 流程的 SQL 里
- 语义模型服务提供标准化 API，各模块（NL→SQL / 报表 / 下游系统）通过 API 获取指标定义
- 指标变更触发下游流程自动刷新，保证全系统指标数值一致性

### 10.1 AI 语义层增强（Tier 1 核心差异化）

| 原方案 | 升级后 | 参考竞品 |
